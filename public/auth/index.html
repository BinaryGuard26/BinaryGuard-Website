Database Security

Supabase / PostgreSQL Controls

Supabase Studio basic authentication is not your production admin security layer. Use PostgreSQL RLS and backend authorization to isolate tenant data.

Rule 1

Every business table needs tenant_id

Orders, dropdowns, pricing, sessions, and audit logs must be tied to the tenant.

Rule 2

Every query restricted by tenant_id

The backend must never return records from another tenant.

Rule 3

Enable RLS before production

Use Row Level Security as a defense-in-depth control for tenant isolation.

Rule 4

Never expose service_role key

The browser must only use safe public keys. Privileged logic belongs in backend API.

Layer 3

Staff / Admin Authorization

Admin access must be separate from client OTP login. Use staff email, password, MFA/OTP, role check, and audit logging.

Email OTP alone is not administrator security

Staff who manage forms, pricing, dropdowns, submissions, tenants, services, or exports must use stronger authentication and role-based access.
